## TL;DR

Run `28705942706` failed because it published `pi-spotify-widget@0.1.6` with the
**pre-fix** `publish.yml` (Node 22, `npm publish --access public` ‚Äî **no
`--provenance`**). The package is configured for **npm Trusted Publishing**, whose
only credential is the OIDC token exchanged when `--provenance` is passed. Without
it, npm received an unauthenticated `PUT` for a package that did not yet exist and
returned **`E404` / `404 Not Found`**. The run that published **`0.1.7`** six
minutes later used the fixed workflow (Node 24 + `--provenance`) and succeeded.
Classification: **Trusted Publishing / authentication** (specifically: publish
invocation missing `--provenance`, compounded by a Node version below npm Trusted
Publishing's supported minimum).

## 4. Cause classification

**Trusted Publishing / authentication** ‚Äî the publish invocation omitted
`--provenance`. Supporting evidence:

1. This package authenticates to npm exclusively via **Trusted Publishing**
   (`docs/release.md`: "uses npm Trusted Publishing with GitHub Actions OIDC";
   "Do not add `NPM_TOKEN`"). The workflow grants `permissions: id-token: write`
   and does **not** set `NODE_AUTH_TOKEN` from a secret. The only way npm receives
   a valid credential is the OIDC token produced by `npm publish --provenance`.
2. The failed run's checked-out `publish.yml` (SHA `c455f47`, identical to the
   `9bc8066` version ‚Äî verified with `git show 9bc8066:.github/workflows/publish.yml`)
   ran `npm publish --access public` **without `--provenance`**, so no OIDC
   exchange occurred. npm then saw an anonymous `PUT` for a non-existent package
   and replied `E404 / 404 Not Found` ‚Äî npm's standard response for "package does
   not exist and you are not authenticated to create it."
3. The diff between the failing and fixed workflow (`git diff 9bc8066 e0ad2ce --
   .github/workflows/publish.yml`) is precisely the Trusted-Publishing repair:

   ```diff
   -        node-version: "22"
   +        node-version: "24"
            registry-url: "https://registry.npmjs.org"
   -        cache: npm
   +        package-manager-cache: false
   ...
   +      - name: Print runtime versions
   +        shell: bash
   +        run: |
   +          node --version
   +          npm --version
   ...
   -        output=$(npm publish --access public 2>&1)
   +        output=$(npm publish --provenance --access public 2>&1)
   ```

   `--provenance` is the operative change; Node 24 is the supported runtime for
   npm Trusted Publishing (the successful `0.1.7` publish ran on `24.18.0`,
   `_npmUser: GitHub Actions <npm-oidc-no-reply@github.com>`).

4. Not a duplicate-version failure: `0.1.6` was never on the registry (¬ß2).
   Not a trigger/configuration failure: the run was dispatched on the correct
   ref (`v0.1.6`) and every preceding step (`npm ci`, `npm run ci`,
   `Skip already published version`) passed; only `Publish to npm` failed.

## 7. Smallest safe correction options

The current workflow is already corrected and publishing works (0.1.7 is live with
provenance). **No code/workflow change is required to publish.** The remaining
gaps are hygiene/hardening only ‚Äî each is a candidate for a **separate,
human-owned correction issue** (npm publication and release actions stay
human-owned). Listed from smallest to largest blast radius:

1. **Publish-time guard against missing `--provenance` / Node < 24 (preferred).**
   Add an early step to `publish.yml` that fails fast with a clear message if
   `setup-node` is not Node ‚â• 24 or if the publish command lacks `--provenance`,
   so a future regression cannot silently reproduce this E404. Workflow-only
   change; no version bump; no publish.
2. **Backfill the missing public versions (0.1.1‚Äì0.1.6) ‚Äî optional, human-owned.**
   These were never published. If version-history continuity on npm is wanted,
   a human can publish the historical tarballs from the respective tags using the
   *current* Trusted-Publishing workflow. **Not recommended unless required**:
   each would need its tag re-dispatched through `publish.yml`, and semver
   consumers are unaffected (0.1.7 already satisfies `^0.1.0`). Capture the
   decision in a separate issue; do **not** attempt here.
3. **Pin `actions/setup-node` to a digest + add a publish dry-run job.** Add a
   non-publish `npm publish --dry-run` step (or a separate CI job) that asserts
   provenance eligibility without uploading. Hardening only.
4. **Add a regression test asserting `publish.yml` contains `--provenance` and
   `node-version: "24"`.** A small `tests/` check (e.g. parse the YAML) would
   prevent the workflow from regressing undetected. Test-only change; no publish.

**Recommended next step:** open one focused correction issue for option 1 (and
optionally 4). Do not edit `publish.yml`, the version, or the CHANGELOG in
DOT-887.
